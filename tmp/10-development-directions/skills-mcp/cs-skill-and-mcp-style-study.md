本文档记录当前 `resources/skills/cs-*` materialized skills 与现有 built-in MCP 的风格，用于后续补齐 OpenBioScience P0 skills/MCP。

## `cs-*` Skill 风格

### 文件结构

典型 `cs-*` skill 是上游 AcademicForge skill 的 OpenScience materialized adapter：

```text
resources/skills/cs-<name>/
  .catalog_stamp
  SKILL.md
  kernel.py            # optional
  provider.py          # optional
  provider.json        # optional
  requirements.lock    # optional
  references/...       # optional
```

### `SKILL.md` 结构

典型结构：

```text
---
name: cs-<name>
description: "<触发场景 + 能力摘要 + OpenScience provenance 提醒>"
---

<!-- Generated by OpenScience science skill materializer; source=... -->

# OpenScience Adapter
...

## How to Read This SKILL.md
...

## OpenScience SOP
...

## Source
...

---

# <Upstream skill body>
...
```

### 共同约束

`cs-*` 的核心不是直接变成可执行工具，而是把上游领域 SOP 接入 OpenScience 运行纪律：

- `openscience-science` 和 `openscience-science-artifact` 是 controlling contract。
- skill 使用本身不能作为 evidence。
- 依赖真实运行、输入、代码、日志、输出、环境、失败记录生成 evidence/provenance。
- 用户可见结果必须进入 `science_artifact`。
- 执行前需要确认 package、service、credential、file、user authorization。
- 上游脚本或模板默认是 source material；若标记为 `quarantined_script`，不能默认执行。

### 常见字段

| 字段/段落 | 作用 |
| --- | --- |
| `name` | Materialized skill id，全部使用 `cs-*` 前缀 |
| `description` | 触发判断入口，通常包含能力边界和 OpenScience provenance 提醒 |
| `Execution policy` | 如 `restricted_default`、`quarantined_script` |
| `risk` | 如 `external_compute`、`credential` |
| `license` | 上游许可 |
| `Source` | 上游 repo、ref、path、original description |
| `OpenScience SOP` | 所有 `cs-*` 的统一 evidence/artifact/provenance 纪律 |

### 对 OpenBioScience 新增 skills 的启发

新增 Bio skills 应避免直接写成“工具说明书”。更合适的结构是：

1. 前半段是 OpenBioScience Adapter：
   - 指向 `openscience-science`、`openscience-singlecell`、`openscience-compute`、`openscience-science-artifact`。
   - 明确 `environmentRef`。
   - 明确 evidence/provenance 义务。
   - 明确哪些 claims 可以说，哪些只能作为 hypothesis。
2. 后半段才是领域 SOP：
   - 输入语义检查。
   - 推荐方法。
   - 最小输出。
   - 常见失败。
   - 与其他 skills/MCP 的交接。

## Built-in MCP 风格

### 代码位置与打包

源码位置：

```text
packages/desktop/src/process/resources/builtinMcp/
```

构建入口：

```text
scripts/build-mcp-servers.js
```

打包产物：

```text
out/main/builtin-mcp-*.js
```

Electron 打包 inclusion 在：

```text
packages/desktop/electron-builder.yml
```

WebUI 会在 `scripts/webui.ts` 中注册 built-in MCP。

### MCP 代码风格

现有 built-in MCP 大多采用：

```ts
const server = new McpServer({
  name: BUILTIN_<NAME>,
  version: '1.0.0',
});

server.tool(
  '<tool_name>',
  '<tool description>',
  {
    action: z.enum([...]),
    ...
  },
  async (args) => {
    ...
    return jsonText({
      schema: '...',
      action,
      ...
      timestamp: Date.now(),
    });
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
```

### 两种 tool surface

| 风格 | 代表 | 适用场景 |
| --- | --- | --- |
| 单 server + 单 control-plane tool + `action` enum | `research_evidence`、`science_artifact`、`lab_skill`、`user_input` | 一个领域有多个相关动作，需要稳定协议和可扩展 action |
| 单 server + 多个小 tool | `medical_evidence` | 每个步骤对 UI/report 有明确语义，且已有稳定命名 |

OpenBioScience P0 更适合第一种风格：新增一个 control-plane MCP，而不是一次注册许多零散 MCP 名称。

### 返回格式

共同特征：

- 返回 JSON text。
- 包含 `schema`、`action`、`timestamp`。
- 和 run/session 相关时包含 `runId` 或 `sessionId`。
- 失败时直接 `throw new Error(...)`，让 MCP 层暴露明确错误。
- 重要结果带 `rawOutput` / `rawPreview` / `structuredContent` / `evidenceDrafts` / `panel` / `object`。
- 输出不直接当最终回答，需要由 `science_artifact` 注册为 evidence、artifact、claim 或 provenance。

### 现有 MCP 能力边界

| MCP | 当前能做 | 当前不能直接替代 |
| --- | --- | --- |
| `research_evidence` | paper/database retrieval；PaperClip 和 `bio_tools` provider dispatch | 本地 matrix/object IO；分析执行 |
| `science_artifact` | artifact graph、evidence、claim、snapshot、publish | 实际运行 Seurat/Scanpy/CellChat 等分析 |
| `medical_evidence` | 医学证据面板、grading、临床证据展示 | tumor response / survival / biomarker 统计 runner |
| `lab_skill` | 从 evidence/conversation 沉淀 skill draft | 运行分析流程 |
| `user_input` | 受控澄清 sample/assay/response/threshold 等关键信息 | 自动判定所有数据语义 |

## 对 P0 Bio Skills/MCP 的方案确认

### Skills 优先补齐

建议先新增或沉淀以下 Bio-specific skills：

| Skill | 类型 | 职责 |
| --- | --- | --- |
| `bio-reproduction` | router | P0 demo case 复现总入口，连接 paper、data、environment、MCP、artifact |
| `bio-singlecell-import` | SOP | 判断 raw counts / processed matrix / metadata / h5ad / Seurat/SCE，并给出安全导入策略 |
| `bio-immune-annotation` | SOP | 免疫细胞大类/小类注释、marker 证据、跨物种注意事项 |
| `bio-signature-scoring` | SOP | CMS-like score、immune state score、response signature、gene set 来源记录 |
| `bio-cci-analysis` | SOP | CCI 分析前置条件、工具选择、结果边界，不把推断交互写成机制因果 |
| `bio-response-comparison` | SOP | responder/non-responder、treatment group、clinical metadata 边界和统计输出 |

这些 skills 应使用 `cs-*` 的 adapter 风格，但来源不是 AcademicForge materialized body，而是 OpenBioScience 自定义 P0 SOP。建议使用 `obs-*` 或 `bio-*` 前缀，避免和 materialized `cs-*` 混淆。

### MCP 优先补齐

建议 P0 先新增一个 built-in MCP：

```text
openscience-bio-runtime
tool: bio_runtime
```

采用单 control-plane tool + `action` enum，而不是拆成多个独立 MCP。

候选 actions：

| Action | 职责 |
| --- | --- |
| `status` | 返回 server-side bio runtime 可用性、official env manifest、已安装 prefix 摘要 |
| `list_environments` | 返回候选 official envs，不把完整包矩阵塞进上下文 |
| `probe_environment` | 对指定 `environmentRef` 做 package/import/CLI smoke |
| `inspect_matrix` | 检查 matrix/h5/h5ad/mtx/metadata 的形态、dtype、整数性、维度、barcode/gene 对齐 |
| `prepare_object` | 生成标准 AnnData/Seurat/SCE 输入对象或对象转换计划 |
| `run_workflow` | 按 workflow id 调用受控脚本，记录 command、cwd、env、log、outputs |
| `summarize_outputs` | 读取 workflow 输出并生成结构化 summary，供 `science_artifact` 注册 |

这个 MCP 负责 object IO 和 analysis runner 的最小受控工具层。`research_evidence` 和 `science_artifact` 继续分别负责文献/数据库证据与 artifact/provenance。

### MCP 设计边界

`bio_runtime` 不应：

- 直接替代 `research_evidence` 做文献检索。
- 直接替代 `science_artifact` 写 artifact graph。
- 让 agent 传入任意 shell 命令并无约束执行。
- 在官方环境里现场安装包。
- 把完整 package version matrix 返回给 LLM。

`bio_runtime` 应该：

- 接受 `environmentRef`、`workflowId`、`inputPaths`、`outputDir`、`parameters`。
- 返回结构化 JSON，包括 `schema`、`action`、`environmentRef`、`command`、`logPath`、`outputPaths`、`warnings`、`timestamp`。
- 把大文件表示为 path/hash/size pointer。
- 明确区分 `supported`、`conditional`、`blocked`。
- 让最终 evidence/artifact 注册交给 `science_artifact`。

## 建议下一步

1. 先设计 `bio_runtime` MCP 的 TypeScript schema 和 action contract。
2. 同步设计 `bio-reproduction` 与 `bio-singlecell-import` 两个首批 skills。
3. 用 `human_CRC` 只验证 `inspect_matrix -> prepare_object -> run_workflow(import_summary)` 的最小链路。
4. 再从该链路沉淀 `immune_annotation`、`signature_scoring`、`cci`。
5. 等 P0a 跑通后，再决定是否拆出独立 MCP 或保持 `bio_runtime` 单 control-plane。
